多主机共同维护博客

根据gemini pro 2.5给出的方案实现，如下

首先建立两个心智模型，hexo博客网站有两个部分组成：

  • 手稿 (Source Files - .md 文件等)
    • 这是创作源头。
    • 使用 git push 来把它备份和同步到您的源文件仓库。
    • 目的：为了在不同电脑间共享您的工作进度，并且进行版本控制。
  • 出版物 (Public Files - .html 文件等)
    • 这是给读者看的最终成品。
    • 您使用 hexo d 来把它发布到您的部署仓库 (GitHub Pages)。
    • 目的：为了更新您线上的网站，让全世界的人都能看到。

所以，实际上是建立两个git仓库来维护，部署仓库就是hexo实际维护的内容，也就是“出版物”，给读者看的东西，而不同主机需要手动共同维护的是“手稿”，是创作的具体内容，比如文章，这部分用另外一个仓库维护。

接下来看看怎么做

下面是这个完整维护方案的全部步骤，从初始设置到日常维护（参考Gemini pro 2.5）

核心思想：双仓库分离

  • 手稿仓库 (Source Repository)：一个私有的 GitHub 仓库，例如 my-blog-source。它用来存放您的博客源文件（.md 文章、主题、配置文件）。我们用 git 命令手动维护它。
  • 部署仓库 (Deployment Repository)：一个公开的 GitHub 仓库，命名必须是 <你的用户名>.github.io。它用来存放最终生成的网站（public 文件夹内容）。由 hexo d 命令自动管理。

第一阶段：在您的第一台主机（主力机 A）上进行初始设置

（假设您已经在主机 A 上建立好了可以正常运行的 Hexo 博客）

  1. 创建手稿仓库

    • 登录 GitHub，创建一个新的、私有的 (Private) 仓库。命名可以随意，例如 my-blog-source。设为私有是为了防止配置文件中的敏感信息泄露。

  2. 关联本地博客目录与手稿仓库

  5. 确认部署配置

    • 打开博客根目录下的 _config.yml 文件，检查 deploy 部分的配置是否指向您的部署仓库 (<username>.github.io)，而不是手稿仓库。
    • 至此，您的主机 A 已经配置完毕。

第二阶段：在您的第二台主机（副机 B）上配置环境

  1. 安装基础环境

    • 确保主机 B 已经安装了 Node.js 和 Git。

  2. 克隆“手稿”仓库

    • 在您想存放博客的任意位置，打开命令行，执行 git clone 命令。注意：是克隆您的手稿仓库，不是部署仓库！
    • 这会把您所有的文章、主题和配置文件下载到主机 B。

  3. 安装博客依赖

    • 进入刚刚克隆下来的文件夹，您会发现里面没有 node_modules 文件夹。
    • 执行 npm install 命令，它会读取 package.json 文件，自动下载所有需要的 Hexo 插件，重建环境。

  4. 验证成功

    • 运行 hexo s 并访问 http://localhost:4000，您应该能看到和主机 A 上一模一样的博客。
    • 至此，主机 B 也已配置完毕，可以随时用于维护博客了。

第三阶段：日常维护的黄金流程

这个流程适用于您的任何一台电脑。请严格遵守，以避免文件冲突。

场景一：开始工作前（比如准备写新文章）

永远先执行这一步，确保你是在最新的版本上工作。

场景二：完成工作后（比如写完了一篇文章、修改了主题）

  1. 第一步：保存你的“手稿”

    • 将您的所有改动（新文章、配置修改等）推送到手稿仓库进行备份和同步。

  2. 第二步：发布你的“出版物”

    • 如果您想让访客看到最新的内容，就执行部署命令。这一步会把网站更新到 <username>.github.io。

只要严格遵循这个“先 git pull，后 git push，再 hexo d”的黄金流程，您就可以在任意电脑间自由切换，无缝地共同维护您的 Hexo 博客了。
