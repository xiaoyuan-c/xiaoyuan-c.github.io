206. 反转链表 - 力扣（LeetCode）

首先想到的暴力解法，其实就是头插法，时间复杂度$\Omicron(n)$

第二种方法，双指针法，让链表直接“掉头”，时间复杂度$\Omicron(n)$

动画206.翻转链表.gif (610×436)纠正：动画应该是先移动pre，再移动cur

注意：针对首元节点的反转，是让它指向nullptr，才让这个节点构成为后来的尾节点，而不是让它指向自己，这样会形成循环（后续看不懂这句话的话直接删除就好，是解释之前直接使用head代替pre，让首元节点在第一个循环中指向了自己，然后导致了错误）

19. 删除链表的倒数第 N 个结点 - 力扣（LeetCode）

初始想法：由于是单链表，所以可以遍历一趟查看有多少个节点，就可以计算出所删除的节点是第几个，再从头遍历到目标节点删除即可，但是这样相当于遍历了两次

进阶想法：使用一次遍历完成。

采用**双指针（快慢指针）**的想法，设置两个指针，slow和fast，均初始化为head节点，先让fast出发，直到fast找到第n个节点，此时slow指向第1个节点，fast指向第n个节点，后续不再让slow停止，而是让fast和slow每次都同时移动一个单位，让slow和fast形成固定的差值，这样当fast到达最后一个节点时，slow自然就是倒数第n个节点。

也就是说，fast先行移动后，slow和fast形成了一个长度固定为n的滑动窗口，右边界fast滑到尾节点的时候，左边界自然就是倒数第n个节点。

但是为了方便删除操作，我们让窗口大小为n+2且设置一个虚拟头节点，

设置长度为n+2的原因：当fast为null的时候，slow就是就是目标节点的前置节点，方便删除

设置虚拟头节点的原因：如果不设置，当n就是链表长度的时候，长度为n+2的窗口会导致空指针错误；设置后，虚拟头节点和最后尾指针指向的nullptr可以看成多出来的两个节点，所以不会造成错误

时间复杂度$\Omicron(n)$

面试题 02.07. 链表相交 - 力扣（LeetCode）

首先想到暴力解法（双指针或快慢指针）：分别遍历两个链表得到各自的长度，得出长度差值，设置两个指针分别指向两个链表，先让长度更长的指针先移动二者的差值个单位，两个指针再同时移动，每次移动后都判断一下是否是同一个节点

时间复杂度为$\Omicron(n+m)$，n是链表headA的长度，m是链表headB的长度

142. 环形链表 II - 力扣（LeetCode）

集合

首先想到用集合收集曾经遍历过的节点，一旦当前遍历的节点出现在集合中，说明这个节点就是入口节点

时间复杂度为$\Omicron(n)$，空间复杂度为$\Omicron(n)$

双指针

第二种想法，双指针法（快慢指针法），用slow和fast判断是否存在环且找到环的入口位置

首先是判断是否存在环，设置慢指针速度为1，快指针速度为2，如果存在环，二者肯定在环内相遇，容易判断是否存在环

那如何找到环的入口位置呢？这需要一点数学推导，如下：

假设从头结点到环形入口节点 的节点数为x。 环形入口节点到 fast指针与slow指针相遇节点 节点数为y。 从相遇节点 再到环形入口节点节点数为 z。 如图所示：

那么相遇时 slow指针走过的节点数为: $x + y$（为什么不是$x+y+n(y+z)$见本题的补充）， fast指针走过的节点数：$x + y + n (y + z)$，n为fast指针在环内走了n圈才遇到slow指针， $（y+z）$为 环内节点的个数。

因为fast指针是一步走2个节点，slow指针一步走1个节点， 所以 fast指针走过的节点数 = slow指针走过的节点数 * 2：
$$
(x + y) * 2 = x + y + n (y + z)
$$
两边消掉一个$(x+y)$:
$$
x + y = n (y + z)
$$
因为要找环形的入口，那么要求的是x，因为x表示 头结点到 环形入口节点的的距离。

所以将x单独放在左面：
$$
x = n (y + z) - y
$$
再从n(y+z)中提出一个(y+z)来，整理公式之后为如下公式：
$$
x = (n - 1) (y + z) + z
$$
其中，n >= 1

这个公式说明什么呢？

  • 先拿n为1的情况来举例

    fast指针在环形里转了一圈之后，就遇到了 slow指针了。

    当 n为1的时候，公式就化解为 x = z，这就意味着，从头结点出发一个指针，从相遇节点 也出发一个指针，这两个指针每次只走一个节点， 那么当这两个指针相遇的时候就是 环形入口的节点。

  • 如果n大于1

    fast指针在环形转n圈之后才遇到 slow指针。

    其实这种情况和n为1的时候 效果是一样的，一样可以通过上述方法找到环形的入口节点，只不过，index1 指针在环里 多转了(n-1)圈，然后再遇到index2，相遇点依然是环形的入口节点。

时间复杂度为$\Omicron(n)$

补充：那么相遇时 slow指针走过的节点数为: $x + y$，为什么不是$x+y+n(y+z)$？为什么slow不能转了几圈再遇到fast呢？

下面证明：slow在它进入环的第1圈内（也就是下图中的环入口2到环入口3）就遇到了fast

先研究一下这种情况：slow和fast在环入口处相遇

环入口1到环入口2的线段可以理解为，fast先进入了环，这是第1圈

环入口2到环入口3的线段可以理解为，fast进入环的第2圈，依此类推

此时fast在第1圈结束的时候，刚好slow进入环，此时二者相遇，这是我们先研究的情况。

如果让slow和fast相遇后继续保持速度不变，继续移动，可以看出，下次相遇，在fast的第2圈结束，slow的第1圈结束，也就是slow走了1圈的时间，fast走了2圈

下面研究普遍情况，在slow进入环的时候，fast在环的某个中间位置，而不是环入口

此时fast距离入口还要走k个节点，slow在环入口2处，这个时候我们记为时刻T1，让slow和fast继续走，直到fast到达环入口3，此时记为时刻T2

从时刻T1到时刻T2，fast共走了$(k+n)$，那么slow走了$\frac{k+n}{2}$

因为时刻T1的时候fast在环内，那k一定小于n，所以$\frac{k+n}{2} < n$，也就是T2时，slow还在环内，在环入口2到环入口3之间

也就是T1到T2的时间内，fast对slow完成了超越，又二者速度只差1，所以二者必定相遇，也即slow在它的第1圈内就遇到fast

证明完毕

总结

上述题目可以看出，处理链表时，经常使用双指针进行操作，思路常常是快慢指针、滑动窗口等
